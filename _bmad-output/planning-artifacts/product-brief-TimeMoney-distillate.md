## Migration Sequencing (Critical)

- **Upgrading from Dart 2.x to 3.11 is a breaking-change cliff, not a smooth upgrade**
- Order matters: SDK constraint must be raised first, but that may break ObjectBox until it is also upgraded
- ObjectBox 1.7→5.x may have schema/API changes — stream APIs (getTimesStream) may work differently
- Freezed 2.x→3.x has breaking codegen changes
- All generated files (.freezed.dart, .g.dart, objectbox.g.dart) must be regenerated at intermediate steps
- **Recommended approach**: define compilation checkpoints where the app must build and run between each upgrade step
- Consider: SDK first → ObjectBox → BLoC → Freezed → remaining deps → architecture restructure → tests

## Architecture Decisions Made

### Freezed Strategy
- **Sealed classes for BLoC events and states** — Dart 3 native, exhaustive pattern matching, no codegen overhead
- **Freezed 3.x retained for domain data classes** (ModelTime, WageHourly) — copyWith, equality, JSON serialization still add value
- **Rationale**: use the right tool for each job; sealed classes where exhaustiveness matters, Freezed where boilerplate reduction matters

### DI Strategy
- **flutter_bloc's RepositoryProvider/BlocProvider** — BLoC-native approach, appropriate for project scale (2 features, 7 use cases, 7 BLoCs)
- **Rejected alternative**: get_it + injectable — over-engineering for this scale, adds codegen dependency
- **Current DI flow preserved but cleaned up**: Repositories → Use Cases → BLoCs via MultiRepositoryProvider → MultiBlocProvider

### Multi-Datasource Strategy
- **ObjectBox** remains primary datasource for mobile/desktop (iOS, Android, Windows)
- **drift (SQLite)** added as web-compatible datasource — uses WebAssembly + OPFS on web
- **Platform-aware DI** selects implementation at runtime based on target platform
- **Both implement the same Repository interfaces** (TimesRepository, WageHourlyRepository)
- **This is the strongest architectural differentiator** — proves Clean Architecture dependency inversion works in practice, not just theory

### flutter_hooks Removal
- Currently used only for `useTextEditingController` in form widgets (CreateHourField, CreateMinutesField, UpdateHourField, UpdateMinutesField, WageHourlyField)
- Replace with standard TextEditingController lifecycle management in StatefulWidget
- **Rationale**: minimal usage doesn't justify the dependency; cleaner portfolio without unnecessary deps

## Architecture Restructure Details

### Current Structure (presentation outside features)
```
lib/src/
├── features/
│   ├── times/ (domain, aplication, infraestructure — NO presentation)
│   └── wage_hourly/ (domain, aplication, infraestructure — NO presentation)
├── presentation/ (ALL UI lives here, separate from features)
│   ├── control_hours/
│   │   ├── times/ (create_time, list_times, update_time, delete_time — each with bloc/)
│   │   ├── wage_hourly/ (fetch_wage, update_wage — each with bloc/)
│   │   └── result_payment/ (cubit/)
│   └── widgets/ (shared widgets)
├── core/ (failures, services, unions, extensions)
└── shared/ (consts, injections)
```

### Target Structure (feature-first)
- Each feature contains ALL layers: domain, application, infrastructure, presentation
- BLoCs live inside their feature, not in a separate presentation tree
- Shared widgets remain in a shared location
- Cross-feature composition (ControlHoursPage that combines times + wage) needs a defined pattern
- Folder spellings corrected: `aplication` → `application`, `infraestructure` → `infrastructure`

### Naming Convention Corrections
- `ITimesObjectboxRepository` → should follow convention where `I` prefixes interfaces, not implementations (currently inverted)
- `TimeBox` / `WageHourlyBox` suffixing with `Box` for ObjectBox entities — evaluate if this convention should extend to drift entities
- Barrel export files need review and cleanup

## Codebase-Specific Details Worth Preserving

### Domain Models
- **ModelTime**: id (int, default 0), hour (int), minutes (int), computed `toDuration`, extension `CalculatePay` on `List<ModelTime>`
- **WageHourly**: id (int, default 0), value (double, default 15.0)
- **Payment formula**: `totalPayment = (totalDuration.inMinutes / 60) * wageHourly`
- No relationship between TimeBox and WageHourlyBox — relationship is implicit at presentation layer

### ObjectBox Service
- Wraps Store and provides Box<TimeBox> and Box<WageHourlyBox>
- Custom stream methods: `getTimesStream()` → `Stream<List<ModelTime>>`, `getWageHourlyStream()` → `Stream<WageHourly>`
- Uses `watch()` for reactive streams — this pattern must be preserved or equivalent in drift
- Separate database names per environment: test-1 (dev), stg-1 (staging), prod-1 (production)

### BLoC Patterns
- All BLoCs follow 4-state pattern: initial, loading, error(GlobalDefaultFailure), success/data
- ListTimesBloc and FetchWageHourlyBloc have 5th state: `hasDataStream` (returns Stream for StreamBuilder)
- CreateTimeBloc and UpdateTimeBloc embed ActionState within composite state alongside form fields
- ResultPaymentCubit is the only Cubit — derives computed state from times list + wage without events
- `Consts.delayed` (500ms Future) used between state transitions — UI animations depend on this timing, evaluate if this pattern is still appropriate

### Error Handling
- GlobalFailure<F>: serverError(F), notConnection, timeOutExceeded, internalError(dynamic, StackTrace?)
- ValueFailure<T>: characterLimitExceeded(T), shortOrNullCharacters(T), invalidFormat(T)
- `GlobalFailure.fromException()` maps SocketException → notConnection, TimeoutException → timeOutExceeded
- Repositories catch all exceptions and wrap in GlobalFailure via `left()` — never throw

### Localization
- EN and ES via ARB files in l10n/arb/
- `context.l10n` convenience accessor
- All user-facing strings must use l10n — never hardcode

### CI/CD
- GitHub Actions using VeryGoodOpenSource reusable workflows @v1 — likely outdated
- Jobs: semantic-pull-request, build (flutter_package), spell-check
- Dependabot configured for daily updates on github-actions and pub ecosystems
- PR template exists with checklist

### Design System (Preserve — No UI Changes)
- Material 3 with colorSchemeSeed: Color(6, 16, 31) (dark blue)
- Delete actions: Red ARGB(255, 163, 70, 64)
- Update/Success actions: Green ARGB(255, 32, 137, 86)
- Wage card: Primary color background with white text
- AlertDialog pattern for all CRUD operations
- ActionState visual feedback pattern on all submit buttons (initial → loading → success → error → auto-reset)

## Rejected Ideas and Decisions

- **DON'T add new features** — this is purely a modernization, features stay identical
- **DON'T change UI/UX** — visual design remains as-is
- **DON'T use get_it/injectable for DI** — over-engineering for project scale
- **DON'T eliminate Freezed entirely** — still valuable for data classes (copyWith, JSON)
- **DON'T add Riverpod** — BLoC is the chosen state management, both are valid but BLoC has stronger enterprise adoption and is the deliberate choice
- **DON'T preserve the `aplication`/`infraestructure` spellings** — original project-context.md said to keep them, but modernization scope explicitly includes correcting these
- **DON'T preserve the I-prefix on implementations** — this inverted convention must be corrected
- **DON'T keep flutter_hooks** — usage too minimal to justify the dependency

## Open Questions for Architecture/PRD Phase

- **Exact target directory tree**: feature-first structure needs concrete mapping of all 90+ presentation files
- **Cross-feature page composition**: ControlHoursPage combines times + wage — where does it live in feature-first structure? Separate "shell" feature? App-level composition?
- **ObjectBox web incompatibility verification**: confirm ObjectBox definitely doesn't support web on Flutter 3.41+
- **drift entity naming convention**: should drift entities also use a suffix (e.g., TimeEntity, TimeDrift)?
- **ObjectBox data migration**: does ObjectBox 5.x maintain backward compatibility with 1.7 database files?
- **Consts.delayed pattern**: is 500ms artificial delay between BLoC states still appropriate, or should it be replaced with proper animation handling?
- **Coverage target**: define specific measurable floor (e.g., 80%+ line coverage on non-generated code)
- **Golden tests**: should screenshot/golden tests be added to prove zero visual regression? Strong portfolio signal but adds complexity
