## Challenges & Growth Areas

1. **"Pre-existing = ignore" pattern** — deferred items classified as "pre-existing" were used as justification to not resolve them. This pattern repeated story after story across Epic 4. Deuda técnica was propagated, not resolved. When drift was implemented, it **replicated** the defects from ObjectBox (create() returning void, silent operations) because the interface already had those defects baked in.

2. **Architectural confirmation bias** — team defended prior decisions as established truth instead of re-evaluating against new context. Concrete case: bootstrap.dart restriction ("do NOT touch") prevented web compilation fix until Christopher escalated. The original decision (no compile-time isolation) was wrong for the multi-platform context, but team resisted changing it.

3. **Specs inheriting invalid restrictions** — Story 4.5 required 3 bad_spec amendments because restrictions were copied from earlier context without validating they still applied. "Do NOT touch bootstrap.dart" and "do NOT modify data layer" were both incorrect for the web platform verification context.

4. **No cleanup story — Epic 3 action item not followed** — despite committing to "every epic includes a final cleanup story" in the Epic 3 retro, Epic 4 was planned without one. Story 4.5 was verification, not cleanup. Deferred items accumulated with no "landing pad."

5. **Developer did not escalate spec vs reality conflict** — when bootstrap.dart restriction blocked web compilation, developer's first response was to defend the restriction rather than propose the correction. Christopher had to detect and escalate the issue.

6. **Story 4.4 with 0 new tests** — DI wiring has no direct unit test coverage. Platform-aware repository selection is verified indirectly through existing tests passing, but no tests explicitly validate the kIsWeb branching logic.

7. **Story 4.5 scope explosion** — originally a "verification only" story, it expanded to include full localization implementation (48 strings), conditional imports refactor, and LocaleCubit creation. While necessary, this scope growth wasn't anticipated in planning.

## Key Insights & Learnings

1. **Prior architectural decisions are hypotheses, not truths** — when context changes (adding web platform), decisions must be re-evaluated. Defending an incorrect pattern because "it was established" is a bias to eliminate.

2. **"No planned resolution" = active technical debt** — if a known problem has no planned fix in any epic, it's not "accepted" — it's unmanaged debt that will compound. The absence of a plan is itself a problem.

3. **Pre-existing ≠ ignorable** — a deferred item tagged "pre-existing" requires either: (a) resolution in the same story, or (b) irrefutable justification + explicit plan for when it will be resolved. "It was already there" is not a justification.

4. **Technical debt infects new code** — drift repositories replicated ObjectBox defects because the shared interface encoded those defects. Debt doesn't stay contained; it propagates through abstractions.

5. **Developer must escalate spec contradictions** — when a spec restriction conflicts with technical reality (compilation failure, correctness issue), the developer should propose the correction proactively, not wait for Project Lead to discover the problem.

6. **Code delivery quality improved, debt management worsened** — patches dropped from 18 (Epic 3) to 1 (Epic 4), showing significant developer growth. But zero deferred items were resolved and no cleanup story was planned, showing regression in debt management.

### Significant Discovery — Conditional Imports Requirement

During Story 4.5, it was discovered that the original bootstrap.dart architecture (runtime kIsWeb branching) was insufficient for web compilation. The web compiler sees all imports at compile-time, including `dart:ffi` through ObjectBox, causing build failure. The solution required conditional imports (`bootstrap_repositories_web.dart` / `bootstrap_repositories_native.dart`) to achieve compile-time platform isolation.

**Impact:** The spec restriction "do NOT touch bootstrap.dart" was incorrect and had to be amended (BS-1). This was only discovered because Christopher tested the web build and escalated the compilation failure. The team's initial response was to defend the existing pattern rather than propose the correction.

**Lesson:** Specs must not inherit restrictions from previous contexts without validation. Build verification on all target platforms should be part of the development workflow, not just the final verification story.
