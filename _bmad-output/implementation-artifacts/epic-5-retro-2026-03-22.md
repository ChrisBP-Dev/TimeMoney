## Challenges & Growth Areas

1. **2 items deferred as "pre-existing" without resolution plan** — Despite Epic 4's action item to eliminate this pattern, Stories 5.2 and 5.3 each deferred one item with "pre-existing" justification. Self-correction worked in 5.1 but discipline was inconsistent in subsequent stories.

2. **Cleanup story absent for second consecutive epic** — Epic 4 retro committed to mandatory cleanup story in every epic. Epic 5 was planned without one. Two epics in a row failing this commitment indicates a systemic gap in planning.

3. **4 bad specs across the epic** — Story 5.1 (2), Story 5.2 (1), Story 5.3 (1). Quality improved story-to-story but spec generation still produces errors requiring developer correction during implementation.

4. **Golden tests don't capture dialog sizing in app context** — Golden tests render widgets directly via `pumpGoldenApp`, not via `showDialog`. This means golden tests verify widget content but NOT how the dialog behaves when opened through the Navigator overlay in the real app. Christopher discovered dialog overflow bugs that goldens didn't catch.

5. **Dialog UI bugs undetected by automated tests** — CreateTimePage dialog expands to full viewport without constraints. This was introduced by pre-Epic 5 quick-dev fix for FractionallySizedBox overflow. Automated tests didn't catch it because they test widget rendering in isolation, not visual behavior in context.

6. **No widget reuse between CreateTimePage and UpdateTimePage** — Both dialogs cover similar functionality (Hour/Minutes fields + action buttons) but were built independently. UpdateTimePage has better structure (Card wrapping) that prevents overflow. No extraction of shared components.

7. **Delete UX anti-pattern** — DeleteTimeButton inside UpdateTimePage dialog deletes immediately without confirmation. Destructive action without confirmation is a UX anti-pattern. Correct pattern: Delete button on TimeCard in list → confirmation dialog with state management.

8. **Code review didn't validate deferred items against retro agreements** — Reviewer accepted "pre-existing" justification without checking it against the Epic 4 agreement that explicitly prohibits this pattern. Reviewer is the last line of defense and didn't fulfill that role.

## Key Insights & Learnings

1. **Auto-correction must be consistent, not selective** — Recognizing an anti-pattern in one story but ignoring it in the next shows the behavior isn't yet habitual. Discipline must be story-to-story, not situational.

2. **Automated tests are a safety net, not a substitute for human verification** — Golden tests verify widget content in isolation. Only human testing in the real app catches contextual issues like dialog sizing, UX flow problems, and visual coherence. Manual verification should be a workflow step.

3. **Golden tests have a blind spot** — Rendering a widget directly via `pumpGoldenApp` does not simulate `showDialog` behavior. Dialog sizing, overlay scope, and Navigator context are not captured. This is a known limitation to document, not a bug to fix.

4. **Widget reuse is a design decision that compounds** — Building CreateTimePage and UpdateTimePage independently was a planning-time decision that propagated inconsistency. When one was fixed (Update), the other wasn't (Create). Shared extraction would have prevented this.

5. **Destructive actions require confirmation** — Delete without confirmation is never acceptable in a professional app. State-managed confirmation dialogs are the minimum bar.

6. **The reviewer is the last line of defense for retro commitments** — If the reviewer doesn't validate deferred items against team agreements, the agreements become aspirational rather than enforced.
